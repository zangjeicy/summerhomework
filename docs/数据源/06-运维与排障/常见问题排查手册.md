# 常见问题排查手册

## 日线数据全部失败

1. 从日志确认识别出的市场。
2. 查看候选数据源和被跳过来源。
3. 区分未配置、运行不可用、短期熔断和请求异常。
4. 检查股票代码是否已转换成供应商格式。
5. 使用对应 Fetcher 的离线测试验证字段处理。
6. 最后再运行有超时限制的网络 smoke test。

## 实时行情为空

- 检查 `ENABLE_REALTIME_QUOTE` 是否开启。
- 检查 `REALTIME_SOURCE_PRIORITY` 的别名是否有效。
- 港股确认是否转换为 `HK` 前缀格式。
- 美股确认密钥型来源是否真的配置。
- 检查是否处于非交易时段，以及供应商是否只返回昨收。
- 不要把 `None` 直接解释为股票不存在。

## 行情看起来是旧数据

- 同时比较行情时间、系统抓取时间和当前交易所时间。
- 确认是否休市、停牌或盘前盘后。
- 检查缓存是否命中。
- 检查 fallback 来源是否更新频率更低。
- 上层应标为 `stale` 并降低结论强度，不能只更新页面显示时间。

## 两个来源数值不同

- 先核对币种、复权、交易时段和时间戳。
- 再核对成交量单位及是否包含盘前盘后。
- 确认一个是实时价还是延迟价。
- 保留差异和来源，不根据期望结果挑选数据。

## 基本面只有部分字段

- 先检查数据包 `status` 是否为 `partial`。
- 不把缺失项填成零。
- 确认市场是否走了正确适配器。
- 检查预算超时和缓存分桶。
- 分析层必须降低置信度或明确说明信息不足。

## 新闻没有结果

- 检查搜索供应商配置，而不是行情 Fetcher。
- 检查查询时间窗口和严格新鲜度。
- 检查公司名、股票代码和市场查询变体。
- 区分没有相关新闻、供应商失败和结果被相关性过滤。

## 排障底线

一次排障结论至少需要日志、配置状态、对应代码路径或可重复测试中的一项直接证据。页面空状态和一次偶发请求都不足以证明根因。
