# 数据源总体架构

## 系统位置

数据源层位于股票代码与配置之后、技术指标和大模型分析之前。它负责把不同供应商的响应转换成系统可消费的数据。

```text
股票代码与市场识别
        ↓
DataFetcherManager 选择数据源
        ↓
具体 Fetcher 获取原始数据
        ↓
Fetcher 标准化与清洗
        ↓
技术指标、基本面上下文、新闻上下文
        ↓
大模型分析、报告与页面
```

## 核心职责

`BaseFetcher` 定义日线数据的共同模板：

- `_fetch_raw_data` 获取供应商原始数据。
- `_normalize_data` 统一为 `date/open/high/low/close/volume/amount/pct_chg`。
- `_clean_data` 转换日期和数值、删除关键空值、按日期排序。
- `_calculate_indicators` 计算 MA5、MA10、MA20 和项目口径的 `volume_ratio`。

`DataFetcherManager` 负责：

- 延迟初始化数据源并按 `priority` 排序。
- 根据市场过滤已知不支持的数据源。
- 日线失败后尝试下一数据源。
- 实时行情按照配置和市场使用独立路由。
- 记录部分运行诊断、健康状态和缓存。

## 四条并行链路

当前代码并不存在覆盖一切的单一返回类型：

- 日线行情主要返回标准化 `pandas.DataFrame`。
- 实时行情由各 Fetcher 返回字典或 `UnifiedRealtimeQuote` 转换结果。
- 基本面由 A 股 AkShare 适配器与港美股 YFinance 适配器形成状态化数据包。
- 新闻由 `SearchService` 调度搜索供应商并做时间、新鲜度与相关性处理。

审查代码时必须分别检查这四条链路，不能用日线接口的结论推断新闻或基本面行为。

## 已实现的稳定性措施

- 单个日线源异常会进入下一个候选源。
- 已知市场不支持的数据源会提前跳过。
- 日线失败达到阈值后会短期熔断，当前阈值为 3 次、冷却 300 秒。
- Fetcher 共享实例的调用由管理器按实例加锁。
- 基本面和部分榜单具备缓存。
- 实时行情优先级可通过配置调整。

## 仍需收敛的设计

- 市场能力表目前内置在管理器，尚未由各 Fetcher 自描述。
- 日线、实时、基本面、新闻的来源元数据结构并不完全一致。
- `None`、空数据、异常和状态包在不同链路中表达方式不同。
- 能力矩阵目前需要从代码审计生成，尚不能直接作为运行时真源。
