# 数据质量状态与来源元数据

## 当前状态词

分析上下文和提示词当前识别以下状态：

- `ok`：数据可正常使用。
- `not_supported`：当前市场或能力不受支持。
- `fallback`：主路径不可用，使用了降级来源。
- `stale`：数据存在但已过期。
- `estimated`：部分内容来自估算。
- `partial`：只有部分字段或区块可用。
- `fetch_failed`：抓取失败。
- `missing`：没有得到所需数据。

这些词已经进入分析上下文约束：当 quote、daily_bars 或 technical 为 `stale`、`fallback`、`missing`、`fetch_failed`、`partial` 或 `estimated` 时，最终置信度不得为高。

## 重要边界

这些状态目前并不是由一个全局枚举贯穿所有 Fetcher。它们在基本面数据包、分析上下文、提示词和诊断结构中已有使用，但日线 `DataFrame` 本身不携带完整质量状态。

因此现阶段应写成“系统分析层识别这些状态”，不能写成“所有数据源统一返回这些状态”。

## 建议保留的来源信息

一次可审计的数据获取至少应保留：

- 请求的股票代码和识别出的市场。
- 请求能力，例如实时行情或日线。
- 尝试顺序。
- 被跳过的数据源及原因。
- 失败数据源的异常类型和摘要。
- 最终数据源。
- 是否发生 fallback。
- 行情时间与抓取时间。
- 是否命中缓存。
- 最终质量状态。

当前 `DataFetcherManager` 已记录部分运行诊断和 fallback 目标，后续优化应复用现有诊断结构，而不是再建一套互不兼容的日志格式。

## 面向 AI 的使用规则

- 降级成功不等于质量与主数据源完全相同。
- 陈旧行情不能被描述成当前实时行情。
- 部分基本面不能推导成完整估值结论。
- 数据缺失时应该降低结论强度，而不是让模型补齐事实。
- 来源冲突应披露差异，不能静默挑选更符合预期的数值。
