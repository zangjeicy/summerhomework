## 5. 系统设置与配置模型

`v0.1.0` 设置 schema 实际包含：基础 1 项、AI 模型 40 项、数据源 27 项、通知 61 项、系统 23 项、Agent 22 项、回测 5 项；动态 LLM 渠道字段会额外加入页面。

配置模型：

- 保存位置：默认项目根 `.env`；若进程显式设置 `ENV_FILE`，则使用该路径。
- 数据库配置：`DATABASE_PATH` 默认 `./data/stock_analysis.db`；认证密码不存数据库，而存数据库目录旁的 `.admin_password_hash`。
- 初始加载优先级：通常进程环境高于 `.env`；Web 保存后以 `setup_env(override=True)` 将 `.env` 重新加载到进程。
- 页面展示优先级：配置页面组合时以保存的 `.env` 覆盖运行时环境 fallback；首次设置状态却让运行时环境覆盖保存值；生成后端廉价状态又以保存值优先。课程应说明不同入口有不同合并语义，不能概括为单一统一优先级。
- 默认值：未保存的 switch 可以展示 schema 默认值；这不代表原始 `.env` 中存在该键。
- 冲突：保存和导入都使用 `mtime_ns + 文件 SHA-256` 形成 `config_version`，不一致返回 409。
- 校验：保存前执行字段校验和跨字段校验；无效请求返回 400/422。
- 写入：优先临时文件 + `os.replace` 原子替换；特定挂载错误时退回原地重写。
- 遮盖：只有 `ALPHASIFT_INSTALL_SPEC`、Hermes 渠道密钥/额外头和 LLM usage HMAC secret 被服务端遮盖；多数注册为敏感的普通 API Key、通知 Token/密码仍以原值进入配置 API 响应。
- 备份：导出的是原始 `.env` 全文，包含秘密；必须当作敏感文件。
- 恢复：导入是“解析键值后合并更新”，不是完整覆盖，也不会删除备份中缺失的现有键。
- 版本迁移：schema 有版本标签，数据库有一条 baseline migration；未发现 `.env` 的通用跨版本迁移框架。导入主要依赖当前 schema 校验和别名兼容，不应宣传为完整版本迁移。

## 7. 首次设置与正常登录

真实状态机：

1. 认证关闭：业务页面和 `/api/v1/*` 默认不要求登录，访问 `/login` 会跳回首页。
2. 认证开启但无密码：访问业务页前端跳转 `/login?redirect=原路径`，登录页显示“设置初始密码”。
3. 首次提交：密码和确认密码一致、至少 6 位；成功后写入哈希文件并设置 Cookie。
4. 已有密码：登录页只显示单密码框；错误返回 401，格式/确认问题通常为 400。
5. 成功登录：写入 HttpOnly、SameSite=Lax 的 `dsa_session` Cookie；HTTPS 或可信转发协议判断为 HTTPS 时增加 Secure。
6. 未登录 API：认证开启时返回 401；前端拦截器跳转登录并保留当前 path + query。
7. 会话过期：Cookie 和服务端签名时间都使用 `ADMIN_SESSION_MAX_AGE_HOURS`，无效/过期返回 401；默认 24 小时。
8. 退出：旋转全局 session secret 并删除当前 Cookie，因此会使所有现有会话同时失效。

现有登录截图对应第 2 种 UI，但状态由 Playwright mock，不是实际启用认证后的截图。没有现成证据覆盖首次密码真正写入、错误密码、成功 Cookie、退出、原地址回跳或过期。

登录页底部固定显示“DSA-V3-TLS”风格文案，但本地截图基于 HTTP。该文案不是 TLS 连接证据，PPT 不得据此宣称 HTTPS/TLS 已部署。

## 8. Cookie、Bearer Token 与管理 API

- 管理 API：认证开启时只检查 `dsa_session` Cookie；没有 `Authorization: Bearer ...` 管理员认证实现。
- Web：Axios `withCredentials=true`，SSE 也携带 Cookie。
- Bearer Token 的真实用途：模型/搜索服务、Slack、ntfy、自定义 Webhook 等出站请求；Discord 使用 Bot Token 头。
- 外部服务 Token 与管理员密码、管理员 Cookie、session secret 完全不同，不能互换。
- PPT 正确说法：**“管理员会话使用 Cookie；Bearer Token 仅出现在部分外部服务调用中，不是本版本管理 API 登录方式。”**

## 9. 权限、限流和网络边界

- 权限模型是单一可选管理员门禁，没有普通用户、角色、权限组或资源级授权。
- 认证开启时，除登录、认证状态、健康检查和 API 文档外，全部 `/api/v1/*` 需要有效 Cookie。
- 认证关闭时，中间件直接放行；这不是“普通用户权限”，而是“无管理认证”。
- 原始 `.env` 导入导出有额外门禁：桌面模式放行；非桌面模式要求认证已开启且 Cookie 有效。认证关闭时返回 403。
- AlphaSift 安装、模型测试、通知测试等没有独立高风险角色，只依赖统一中间件；认证关闭时同样无登录门禁。
- 登录限流：单进程内存字典，以客户端 IP 为桶；5 次失败后限制 300 秒，返回 429；成功后清除该 IP 桶。
- 限流不是跨进程、跨实例或持久化方案；重启会清空。
- `TRUST_X_FORWARDED_FOR=true` 时取 `X-Forwarded-For` 最右侧地址，修复了简单的左侧伪造绕过，但没有可信代理 IP allowlist，也不验证请求是否真的来自反向代理。
- 当前教学配置未显式启用可信转发。部署到局域网、公网或多级代理时必须单独设计代理清洗、可信跳数、TLS 和网络 ACL，不能只打开该开关。

## 10. 密码修改与恢复

- 页面内修改密码：存在；要求当前密码、新密码和确认密码，统一门禁下需要有效 Cookie。
- 修改密码后：只替换密码哈希，没有旋转 session secret，已有会话不会立即失效。
- 忘记密码：没有 Web 忘记密码页面、邮件找回、恢复码或管理 API。
- CLI 重置：`python -m src.auth reset_password`；交互式隐藏输入，要求认证已在 `.env` 开启，直接覆盖密码哈希。命令已由代码和测试确认，本任务未执行。

## 13. 4xx、5xx 与合法空状态

| 状态 | 本版本语义示例 | 自动验收处理 |
| --- | --- | --- |
| 400 | 密码/配置/日期参数错误、导入内容无效 | 默认失败 |
| 401 | 未登录、密码错误 | 默认失败；前端通常跳登录 |
| 403 | 非桌面且认证关闭时访问原始 `.env` 备份 | 默认失败 |
| 404 | 回测汇总不存在；前端转换为 `null` 并显示空状态 | 仅明确 allowlist 的回测 404 被允许 |
| 409 | 配置版本冲突 | 默认失败，要求刷新后重试 |
| 422 | 请求结构或测试参数无效 | 默认失败 |
| 429 | 登录失败次数过多；外部新闻源也可能返回 429 | 页面验收默认失败；外部日志需按供应商失败处理 |
| 500 | 未捕获服务端错误或外部服务失败被转成服务错误 | 默认失败 |

回测 404 的真实含义是“数据库中没有匹配的回测汇总”，不是路由不存在，也不是回测成功。自动验收的 allowlist 是产品契约声明，不会把 404 变成业务成功。

## 14. SHA-256 的证据边界

SHA-256 可证明：

- 当前 ZIP 内容与已记录哈希一致；
- 教学目录某文件与 ZIP entry 内容一致；
- 当前截图文件与验收 JSON 记录的截图字节一致。

SHA-256 不能证明：

- 截图由哪个 commit、配置、数据库或登录状态生成；
- 截图中的数据不是 Mock、没有污染或来自实时接口；
- 页面业务操作成功；
- 数据、模型结论、收益或回测结果正确；
- TLS、权限或部署安全已经成立。
