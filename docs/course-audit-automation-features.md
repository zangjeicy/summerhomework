## 5. 首页分析任务生命周期

```text
输入 → 前后端校验 → POST /analysis/analyze
  ├─ 非法输入/市场/股票 → 4xx
  ├─ 同股票正在处理 → 单只 409；批量跳过重复项
  └─ 接受 → pending → processing → completed
                              └→ failed
  cancel_requested/cancelled 仅有模型/UI，无实际取消入口
```

- 后端解析和规范化市场代码，批量最多 50；同步只允许单股票，异步支持批量。
- 单只重复在途任务返回 409；批量返回已接受和被跳过项。去重键只存在于当前进程的 `_analyzing_stocks`。
- 队列使用线程池；进度单调更新，完成为 100。提交 executor 失败时会回滚批次内存状态。
- 分析完成后保存报告、历史和信号；已完成报告不阻止再次提交。`force_refresh` 是流水线刷新参数，不是任务结果缓存锁。
- 失败状态和错误广播已实现，但没有教学环境真实失败样本。
- 取消状态枚举、schema 和前端识别存在，却没有 cancel API 或队列取消方法，属于部分实现。
- 分析历史持久化；任务历史仅在内存并按上限清理。

## 6. 进度更新与任务恢复

首页采用 SSE 为主、30 秒轮询校准为辅；没有 WebSocket。

- SSE 事件：`connected`、`task_created`、`task_started`、`task_progress`、`task_completed`、`task_failed`、`heartbeat`；heartbeat 30 秒。
- 前端断线后约 3 秒重连；重连/页面重新可见时刷新活动任务。
- 无 `Last-Event-ID`、持久事件日志或游标重放；新连接只补发当前进程的活动任务快照。
- AlphaSift 不用 SSE，而是每 2 秒轮询，并把活动任务 ID 存于 `sessionStorage`。

| 场景 | 行为 |
| --- | --- |
| 浏览器短断线/刷新 | 可重连并从当前进程活动任务校准 |
| 应用重启 | `_tasks/_futures/_analyzing_stocks` 全丢失，无在途恢复 |
| 已完成报告 | 历史和报告仍在；部分查询可按 query_id 找到完成结果，但不是任务恢复 |
| 失败重试 | 无通用自动重试；锁释放后用户可重提 |
| 多实例 | 任务和去重均不共享，不能跨实例互斥 |

课程可讲“断线重连和轮询校准”，不能讲“可靠消息队列、断点续跑、跨服务恢复、失败自动重试”。

## 9. 告警规则与调度

v0.1.0 明确支持：

| 类别 | 规则 |
| --- | --- |
| 行情 | `price_cross`、`price_change_percent`、`volume_spike` |
| 技术 | `ma_price_cross`、`rsi_threshold`、`macd_cross`、`kdj_cross`、`cci_threshold` |
| 组合 | `portfolio_stop_loss`、`portfolio_concentration`、`portfolio_drawdown`、`portfolio_price_stale` |
| 市场 | `market_light_status`、`market_light_score_drop` |

范围包括单股票、自选股、持仓、账户组合和市场；支持阈值/方向、info/warning/critical、启用、冷却及 dry-run 测试。dry-run 不写触发历史、不发送通知。不能把明确规则泛化为任意均线、MACD 柱、KDJ 超买超卖或自定义规则。

worker 逐条隔离评估，可记录 `triggered`、`skipped`、`degraded`、`failed`；未命中通常不产生同等持久记录。数据库规则冷却可持久化。

调度由 Web lifespan 启停，CLI 另有纯调度模式；每日按进程本地时间运行，后台循环约 30 秒检查，事件监控默认 5 分钟。互斥锁是进程内 lock，无分布式锁；项目关闭后 daemon thread 停止。失败记录 last_error，无持久队列/通用退避重试。教学配置关闭调度和事件监控，日志无实际调度证据。

## 10. 通知、失败与重试

通知支持飞书、钉钉、企业微信/微信类渠道、Telegram、邮件、Discord、Slack、PushPlus、ServerChan、Pushover、ntfy、Gotify、自定义 Webhook 等。每渠道结果包含 success、error_code、retryable、latency 和 diagnostics。

告警 worker 先保存触发，再逐渠道保存通知尝试，因此“命中”和“通知成功”分开记录；通知失败不抹掉命中，也不阻止其他规则。`retryable` 只是诊断，不代表统一持久化重试队列；少数 sender 对特定错误有有限重试。设置页测试通知会真实外发，本轮未调用。

教学日志显示没有有效通知渠道，数据库通知记录为 0；报告保存成功不等于通知成功。分类：已实现但未完成业务验收。
