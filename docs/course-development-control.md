本文定义 RuyiDailyStockAnalysis 课程、项目 docs、问题、`dev` 开发线和累积式课程版本之间的最小闭环。它不替代根 `AGENTS.md`，也不替代 RuyiTypora 任务看板。

| 层 | 真相内容 | 入口 |
| --- | --- | --- |

两边使用同一个 `RDSA-...` 任务编号或 `RDSA-COURSE-ISSUE-...` 问题编号互相引用，不复制完整技术报告。

## 课程与代码真相源

- 当前课程版本是 `v0.1.0`，PPT 01、02、03 都属于该版本，可以逐步完成。
- `RuyiDailyStockAnalysis_v0.1.0.zip` 是 `v0.1.0` 教学代码的第一真相源。
- 当前开发仓库 `dev` 只用于代码演进对照、问题修复参考和下一版本候选来源。
- 不得把 `dev` 中尚未进入稳定教学 ZIP 的行为写成 `v0.1.0` 已有功能。
- 主课程 PPT 是课程内容唯一真相源；院校或企业授课版只是轻量品牌副本。

## 当前版本讲解原则

1. 当前版本必须按教学 ZIP 的真实实现、页面状态、配置和计算口径讲解。
2. `v0.1.0` 的已知限制、空状态、未启用功能和疑似问题可以进入 PPT 与讲师备注，但必须准确标记状态。
3. 发现问题不自动重写、覆盖或否定历史课程版本。
4. 当前 PPT 可以说明“当前版本限制”“下一版本计划”“下一版本预览”，但不得混淆为现有能力。
5. 尚未核验的问题只能写“待核验”，不能写成确认缺陷或承诺具体修复版本。

## 问题反哺与增量课程闭环

1. 在 PPT 制作、讲师备课、授课或学生使用中发现异常，先在 [课程问题索引](course-issues.md)去重并登记发现版本。
2. 007～010 以 `v0.1.0` ZIP 为第一真相源核验当前行为，再与 `dev` 做差异对照。
3. 已确认且需要代码修改的问题进入 `dev`；不得直接修改 `main`，也不得反写成 `v0.1.0` 已经修复。
4. 修复完成后依次保留自动测试、API/页面验证、课程验证和文档证据，作为 `v0.1.1` 或后续版本候选。
5. 下一版本课程必须讲清：改了什么、为什么修改、如何修改、如何测试、页面和架构发生了什么变化。
6. 产品问题、修复过程、测试方法和版本升级本身可以成为增量课程内容。

## 编号和状态

- 历史项目任务继续使用 `RDS-xxx`，永不改号。
- 课程任务使用 `RDSA-<模块>-<任务类型>-<序号>`。
- 课程问题使用 `RDSA-COURSE-ISSUE-<序号>`。

任务状态：

`待讨论 → 方案已确定 → 待执行 → 执行中 → 等待汇报 → 待验收 → 发现问题 → 待修复 → dev 修复中 → 待复验 → 已完成 → 已归档`

问题生命周期：

`待核验 → 已确认 → 待设计修复方案 → dev 修复中 → 自动测试通过 → 页面验证通过 → 课程验证通过 → 文档已更新 → PPT 已更新 → 待合并主分支 → 已关闭`

## PPT 影响评估

下列变化必须判断影响哪个课程版本、PPT、讲师备注和学生代码包：

- 页面、导航、字段、空状态、错误状态或认证流程变化。
- CLI、API、配置名、默认值、启动方式、部署或依赖变化。
- 计算口径、数据来源、fallback、Token 统计、任务状态或报告语义变化。
- 演示步骤、截图、测试期望或复现数据变化。

纯内部重构且用户契约和教学结果均不变时，可以不更新 PPT，但需要在版本记录中保留“不影响课程”的依据。
